# 파일을 더 잘 읽는 방법: 코딩 에이전트를 위한 솔루션

> 출처: https://blog.sweep.dev/posts/read-file
> 작성자: Veronica | 2026년 1월 30일

## 개요

코딩 에이전트가 대규모 파일을 읽을 때 발생하는 토큰 낭비 문제를 해결하는 방법을 제시한다.
핵심 성과는 **초대형 파일의 토큰 사용량을 90% 감소**시킨 것이다.

---

## 1. 문제 정의

### 1.1 기존 라인 기반 파일 읽기의 한계

코딩 에이전트가 대규모 파일(예: 5,000줄)을 읽을 때, 전체 내용 처리에 **40,000개 이상의 토큰**이 소모된다.
이는 높은 비용뿐 아니라 지연 시간 증가와 **"컨텍스트 부패(context corruption)"** 문제를 야기한다.

기존 인터페이스 구조:

```json
{
  "tool": "read_file",
  "parameters": {
    "path": "src/services/order_processor.py",
    "start_line": 100,
    "end_line": 150
  }
}
```

핵심 문제점:
- 에이전트가 파일 섹션의 내용을 **보기 전에** 관련성을 판단해야 한다
- **과다 읽기**(토큰 낭비) 또는 **부족 읽기**(재요청 필요)의 양극단에 빠진다

### 1.2 추가 도구 활용의 한계

"파일 개요(file outline)" 도구를 별도로 제공하는 시도도 실패했다.
이유는 에이전트가 개요를 먼저 확인하는 방식으로 학습되지 않았기 때문으로,
모델의 기본 행동 성향에 역행하기 때문이다.

## 2. 제안된 솔루션

### 2.1 핵심 아이디어

기존 `read_file` 도구를 래핑하여, **파일 크기에 따라 자동으로 응답 형식을 결정**한다:

| 파일 크기 | 반환 형식 |
|-----------|-----------|
| 소형 파일 | 전체 내용 반환 |
| 대형 파일 | 구조 개요 반환 (라인 번호 포함) |

이 방식은 모델의 학습 방식과 조화를 이루며, 특별한 프롬프팅 없이 자연스럽게 작동한다.

### 2.2 토큰 효율적인 개요 형식

비효율적 형식 (15 토큰):
```
[...5 nested items, read lines 298-558 for details]
```

최적화된 형식 (9 토큰):
```
(5 children) [298:558]
```

약 **30% 토큰 절감**을 달성했다.

### 2.3 완전한 개요 형식 예시

```
# File Structure Outline
[public] class MyClass [1:150]
  [private] fun initialize [10:25]
  [public] property config [30]
  [protected] fun processData (3 children) [40:95]
  [public] fun getResults [100:145]
```

각 항목에는 다음 정보가 포함된다:
- **접근 제어자**: `[public]`, `[private]`, `[protected]`
- **심볼 종류 및 이름**: `class MyClass`, `fun initialize` 등
- **자식 수**: `(3 children)` — 축약된 하위 항목이 있을 때 표시
- **라인 범위**: `[시작:끝]` — 해당 심볼이 위치한 라인 번호

### 2.4 적응형 깊이 조정 (Adaptive Depth)

매우 큰 파일의 경우 동적으로 깊이를 제어한다:

1. 무제한 깊이로 개요를 생성한다
2. 크기를 확인한다 (10,000 토큰 이상이면 깊이를 단계적으로 감소)
3. 깊이 1(최상위 기호만 표시)까지 반복한다

예시:
```
깊이 무제한: 모든 중첩 항목 표시
깊이 2:     중첩 클래스 축약
깊이 1:     최상위 심볼만 표시
```

---

## 4. 핵심 인사이트

이 접근법의 강점:

1. **모델 친화적**: 에이전트가 특별한 학습 없이도 효율적으로 대형 파일을 탐색할 수 있다
2. **정밀한 탐색**: 에이전트가 관련 섹션을 정확히 식별하고 필요한 부분만 요청한다
3. **투명한 동작**: 기존 `read_file` 도구를 그대로 사용하므로 에이전트 워크플로우 변경이 불필요하다
4. **점진적 상세화**: 개요 → 관심 영역 → 상세 코드 순으로 자연스럽게 drill-down한다
